At Your Disposal

It's easy to get lost during an extended interaction with an LLM.

We start out enthusiastically but after much scrabbling around and changes of direction we reach a dead end surrounded by the skeletons of unworkable solutions.

One response is to augment the LLM by increasing its context window. The alternative is to bolster the human.

We do this by structuring and documenting human-LLM interactions.

We have chosen the metaphor of Ariadne's Thread to describe our approach to solving this problem. Ariadne handed the Greek hero Theseus a ball of thread before he entered the labyrinth at Knossos so that he could retrace his steps after confronting, and killing, the Minotaur.

Escaping the labyrinth

What we want

For the purpose of this discussion, we are going to focus on creating a project prototype with the help of an LLM. We expect this kind of interaction to generate quite a bit of back and forth between the developer and the LLM.

  • A place for the developer to think outside the interaction
  • A set of guidelines or rules for the LLM to follow during the interaction
  • A way to document the interaction

How to get it

Since we are creating a project, we can create a space inside the project to document the interaction. Our project outline and interaction rules go there, a record of the interaction and any supporting materials.

To see what this looks like in practice, take a look at the artefacts directory for a project prototype that became an npm package we use for generating rss feeds.

We've created a tool to help us with this setup, the project prototype generator . As part of your project setup, this tool will create a thread with a prepopulated prompt template and an empty snapshots array. All you have to do is fill in the project definition (and there are human prompts to guide you).

We have a prompt template to help us. It consists of two parts:

  • Thread definition

    This is where we put project specific items: the project title, description, the prototype type, and details of what we want to do.

  • LLM instructions

    This is where we put general instructions such as whether the LLM should show code, ask questions when in doubt, and whether to be succinct or verbose.

    We also define how the LLM should summarise the interaction to date. We call this a snapshot. In this case, we provide a snapshot schema in JSON but it could be a markdown template or free text.

Once we have completed the prompt template, we use it to initialise our interaction by adding or uploading the file. Our interaction is now a thread and we can start writing prompts.

At any point during the interaction, we can ask the LLM to generate a snapshot. We do this when we detect a discernible shift in thinking or approach, or when we want to review what we have done. We add these snapshots to a thread history.

Thread workflow

See a state diagram of a thread workflow.

In summary, we embellish or decorate an interaction by adding a thread which defines what we want to do and what form the interaction should take. The thread template also defines a snapshot (the summary the LLM periodically generates at our request).

We can now:

  • Capture progress (decisions made, etc.)
  • Ask for a summary of recent interactions
  • Go back in the thread to a previous snapshot
  • Review the interaction to date (thread history)
  • Break down unwieldy interactions into manageable parts
  • Penetrate further into the labyrinth without fear of pitfalls

See the Ariadne's Thread project repo README for hints and tips.

Conclusion

LLMs are excellent at small, discrete tasks. They are typically reliable at convergent thinking (refining ideas) and with a little oversight can help speed things along.

They've helped me expand my ambitions by plugging gaps in my knowledge (handling unfamiliar languages or APIs) and doing some of the heavy lifting (writing tests, generating diagrams).

They are endlessly patient but they are not cheap. Their cost is paid in finite resources such as water, pollution and emissions as well as money.

Ariadne's Thread

Ariadne's Thread emerged from two different but complementary wishes.

  • To be independent of LLMs (because of their cost)
  • To document interactions outside of the chat interface so as to avoid getting lost

Dependency avoidance

I use LLMs (quite a lot) but I don't want to be reliant on them. Where possible I create something new from my interactions, something which does not require their continued presence. I think LLMs are valuable but not indispensable.

The work on Ariadne's Thread is itself an example of this principle in practice; I worked with three LLMs to test its viability but the solution is not dependent on having access to an LLM.

Critical thinking

Over the past couple of years I've adapted to using LLMs and trained myself to write concise prompts that specify what I want, what I don't want, and how I want the LLM to respond.

Converting an interaction into a thread extends this practice. I proceed from prompt to prompt with a definite end in mind and when forced to change direction or retrace my steps, I capture progress to date in a snapshot.

By putting more effort and thought into my interactions, I use LLMs less frequently and with better results. And there's something more, I increasingly answer my own questions without recourse to an LLM.

Terms

Thread
The record of an interaction between a developer and an LLM. Contains the initial prompt template and a collection of snapshots.
Prompt template
Used to initialise a thread. Contains project-specific details, rules on how an LLM should interact and how it should format periodic summaries (snapshots).
Snapshot
A specific interaction state representative of progress to date. A snapshot should conform to the snapshot schema.
