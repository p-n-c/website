Escaping the labyrinth

People and Code
At Your Disposal

When working on projects with an LLM, I often became lost, unsure how to proceed, and too far from my starting point to retrace my steps.

As I wondered what to do about this, I was reminded of the ball of thread Ariadne gave to Theseus. I too needed a thread to guide me.

Ariadne's Thread

In Greek mythology, the Cretan princess, Ariadne handed Theseus a ball of thread before he entered the labyrinth at Knossos so that that he might find his way back to safety after confronting, and killing, the Minotaur.

I'm going to jump straight into the problem, how not to get lost when interacting with an LLM. I'll use the example of creating a project prototype because prototypes are sufficiently complex to require a bit of back and forth between a developer and an LLM.

I'll describe each part of the problem, and it's solution, in turn. At the end, we'll see how they fit together.

The project

We need to know what we're going to be talking about.

LLMs like to attach names to things so I prefer to name my project upfront. It also helps to give it a short description and a more detailed definition. This will benefit me as much as the LLM.

Prompt template: Used to initialise a thread. Contains project-specific details, rules on how an LLM should interact and how it should format periodic summaries (snapshots).

This is the kind of information that typically goes in the initial prompt but I'm calling it a prompt template because we will use its format repeatedly. In addition to the project title, description and definition, I add a prototype type and its definition. A conceptual prototype is different than a vertical or functional prototype and this helps me sort out my thinking before I set out.

The thread

Thread: The record of an interaction between a developer and an LLM. Contains the initial prompt template and a collection of snapshots.

The thread is what is going to guide me and enable me to document my interaction with an LLM, or LLMs.

My thread is written in JSON. It's a simply structured record of the interaction. Here's a basic example:

thread.json

              {        
                thread: {
                  promptTemplate: {
                    project: {
                      title: 'project title',
                      description: 'project description',
                      definition: 'project definition',
                    },
                    prototype: {
                      type: 'prototype',
                      definition: 'prototype definition',
                    },
                  },
                },
              }

I'm going to set some ground rules or guidelines for the LLM in my prompt. By default I include an explicit direction: "The LLM should follow these guidelines throughout". Rules govern things like whether the LLM should show code by default.

These are particularly effective for pacing the LLM's response:

  • confirmBeforeAdvancing
  • explainOneConceptAtATime

They cause the LLM to break solutions down into steps and iterate over them.

It's a good idea to remind the LLM of these rules as you go along and you can override them at any time. (NB They are not prescriptive; tailor them to your needs).

Here's my updated thread. The prompt template has been fleshed out, and a new field, snapshots, added:

thread.json (with rules, terms and snapshots) 

              {
                thread: {
                  promptTemplate: {
                    project: {
                      title: 'project title',
                      description: 'project description',
                      definition: 'project definition',
                    },
                    prototype: {
                      type: 'prototype',
                      definition: 'prototype definition',
                    },
                    interactionGuidelines: {
                      description: 'The LLM should follow these guidelines throughout.',
                      rules: {
                        conciseByDefault: true,
                        showCodeByDefault: false,
                      },
                      terms: {
                        promptTemplate: {
                          definition: 'Used to initialise a thread. Contains…',
                        },
                      },
                    },
                  },
                  snapshots: [],
                },
              }

The snapshot

Snapshot: A specific interaction state representative of progress to date. A snapshot should conform to the snapshot schema.

The snapshot is how I keep track of what's going on.

Whenever I want a summary of the interaction to date, need to slow things down, or want to record where I am before changing direction, I ask the LLM for a snapshot.

I choose to keep a record of snapshots by copying and pasting them into the snapshots array in my thread. You might store want to share them with others or have LLM curate them.

The snapshot schema maintains consistency and it's primarily for my benefit, not the LLM's. I've written it in a form that I find easy to parse, but whether it's in yaml or markdown won't make much difference to the LLM.

The snapshot schema
                {
                  schema: {
                    project: {
                      title: 'string',
                      description: 'string',
                      definition: 'string',
                    },
                    prototype: {
                      type: 'string',
                      definition: 'string',
                    },
                    summary: 'string',
                  },
                }

You'll notice the snapshot has a copy of the project and prototype objects. This reminds the developer and the LLM of the purpose of the chat.

Artefacts

I now have a thread which contains my initial prompt, interaction guidelines, and a snapshots array for tracking progress. Where should I put it?

It makes sense to keep the thread in the project but separate from the source code. I add the thread to a new directory, artefacts, which is where I'll also put files used during the interaction as well as code or designs generated by the LLM.

I can now step away from the chat interface and look over the interaction in my own time. It feels much calmer. I have time to review code, research any dependencies introduced, and think about what I want to do next. It also means I don't copy and paste directly into my source code from the chat.

The artefacts directory

                artefacts: 
                  thread.json
                  // uploaded files and examples
                  // responses provided by the LLM
                src: 
                  // project code

Workflow

The Ariadne's Thread repo has a visual representation of the workflow.

How does this work in practice? Here's a step-by-step guide for creating a thread for a small project or prototype:

  • Create a new project
  • Add an artefacts directory
  • Create a new thread file in artefacts with prompt details, interaction rules and how you want the LLM to write snapshots
  • Open a new chat and upload the thread
  • Add any files used during the interaction to artefacts, including those generated by the LLM
  • When you need a summary of progress to date, ask for a snapshot and add it to the snapshots array
  • Use the snapshots and LLM responses inside artefacts to think over what you are doing before working on source code and away from the distractions of a chat interface

Example - Building a simple RSS feed generator with Claude

I used Ariadne's Thread to create a simple RSS feed generator. In the artefacts folder you will find a thread, an example of an RSS feed which I added to the chat, and an rss-generator file with code from the LLM. Because I had asked Claude to proceed incrementally, it returned small chunks of code (numbered 1-4).

Conclusion

In order not to get lost in the maze of dead-ends and wrong turns caused by muddled thinking, I struck on the metaphor of the thread which Ariadne gave to Theseus to help him navigate the labyrinth.

Creating a thread has the effect of making me think more carefully before starting a new project. Its structure and documentation afford me better oversight and - if I ask an LLM to assist me - the means to moderate our interaction.

The project prototype generator expedites creating a prototype that supports Ariadne's Thread. When you create a new prototype, the artefacts folder and a thread complete with prompt template is included. A list of human prompts will help you define your project.
